10 年職人手記：如何運用 Figma 與 Storybook 打造美感與邏輯兼具的企業級設計系統

設計系統讓設計語言、工程元件與團隊協作有共同標準，降低跨部門溝通成本。

設計系統真正困難的地方，不是做出漂亮的元件庫，而是讓它在團隊日常中持續被使用。Figma 和 Storybook 的搭配，能讓視覺規格、工程狀態與協作文件保持在同一條線上。

先定義 token，再討論元件

很多團隊一開始急著整理 Button、Card、Modal，卻忽略顏色、字級、間距與狀態這些基礎規則。先把 token 命名清楚，元件才不會在實作時長出一堆例外。

Figma 負責視覺語言，Storybook 負責真實狀態

Figma 適合定義元件外觀和使用情境，但產品裡還會遇到長文字、空資料、載入中、錯誤與 disabled 等狀態。Storybook 能把這些狀態變成可互動文件，讓討論更接近真實產品。

命名規則決定維護成本

顏色不應只叫 orange 或 gray，而應對應 primary、neutral、danger 等語意。文字階層也應用 display、heading、paragraph、label 這類用途命名，讓設計稿和 class 更容易對齊。

設計系統分層交付對照表

系統層級主要工具交付重點
Design TokensFigma Variables / CSS Variables色彩、字級、間距與陰影命名
ComponentsFigma Components / HTML 組件可重用元件與互動狀態
DocumentationStorybook / 使用規範使用情境、限制與狀態展示
GovernanceReview 流程新增、修改與淘汰規則

文件化不是額外工作，而是交付的一部分

企業級設計系統需要清楚說明何時使用、何時避免，以及元件有哪些可調整屬性。這些文件能讓新成員快速進入狀況，也能降低跨團隊溝通成本。

設計系統不是把畫面做整齊，而是讓團隊在不同需求中仍然能做出一致決策。

設計系統不會第一次就完美。它需要被真實專案反覆驗證，再從使用情境中修正。當設計與工程共同維護它，它才會成為產品長期成長的基礎。
